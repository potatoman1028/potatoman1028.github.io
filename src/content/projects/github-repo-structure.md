# GitHub 레포 구성도

이 문서는 현재 내가 운영하고 있는 `Potato` 계열 저장소를 어떤 기준으로 나누고 있는지 정리한 기록이다.

처음부터 이렇게 나눠둔 건 아니고, 이것저것 한 저장소에 몰아넣다 보니 학습용 코드와 실제로 유지할 코드, 그리고 C# 포팅 실험 코드가 서로 성격이 너무 달라서 관리가 애매해지는 순간이 왔다.

그래서 지금은 역할을 기준으로 저장소를 분리해서 운영하고 있다.

![GitHub 레포 구성도](/projects/github-repo-structure.png)

## 저장소를 나눈 기준

이번 구조에서 중요하게 본 건 "무슨 언어를 쓰느냐"보다 "어떤 성격의 작업이냐"였다.

정리하면 아래 세 가지로 나뉜다.

| 레포지토리 | 역할 | 기본 브랜치 운영 |
| :--- | :--- | :--- |
| **Potato.Study** | 개인 학습, 샘플, 실험 기록 | `main` |
| **Potato** | C++ 메인 라이브러리와 실사용 코드 | `main`, `dev` |
| **Potato.CSharp** | C# 포팅 및 빠른 실험 | `main`, `vibe` |

즉, 학습은 학습대로 남기고, 실제로 계속 가져갈 코드는 따로 관리하고, 포팅 실험은 또 별도 공간에서 다루는 방식이다.

## 각 저장소의 역할

### `Potato.Study`

여기는 말 그대로 공부하고 기록하는 공간이다.

새로운 아이디어를 시험해보거나, 작은 샘플을 만들어보거나, 아직 구조를 진지하게 가져갈 단계가 아닌 코드를 모아두는 용도에 가깝다.

정리되지 않은 생각이나 시행착오도 여기서는 괜찮다. 대신 이 저장소의 결과물이 바로 실사용 코드로 이어진다고 가정하지는 않는다.

### `Potato`

이 저장소는 실제로 유지하고 싶은 C++ 메인 라이브러리를 두는 곳이다.

여기서는 학습용 코드보다 안정성과 누적 가능한 구조가 더 중요하다. 그래서 브랜치도 `main`과 `dev`를 분리해서 운영한다.

- `main`: 실제 사용 기준의 안정 브랜치
- `dev`: 통합과 검증을 진행하는 브랜치

즉, 뭔가를 빨리 시험해보는 공간이라기보다, 계속 다듬어 가면서 남길 코드를 관리하는 쪽에 가깝다.

### `Potato.CSharp`

여기는 C# 포팅과 관련된 실험 공간이다.

C++ 쪽 구현을 그대로 옮기는 작업도 있고, 포팅 과정에서 구조를 다시 생각해보는 경우도 있어서 메인 저장소와는 결이 조금 다르다. 그래서 실험 속도를 우선하는 `vibe` 브랜치를 따로 두고 있다.

- `vibe`: 빠르게 옮겨보고 감을 잡는 실험 브랜치
- `main`: 어느 정도 정리된 결과를 보관하는 브랜치

결국 이 저장소는 "완성된 제품"보다는, C# 쪽 방향성을 확인하고 축적해보는 공간에 더 가깝다.

## 현재 운영 흐름

전체 흐름은 대략 아래 순서로 보고 있다.

1. `Potato.Study`에서 아이디어나 샘플, 학습 내용을 먼저 정리한다.
2. 실사용 가치가 있다고 판단되는 내용만 `Potato/dev`로 가져와 개발하고 검증한다.
3. C#으로 옮겨볼 가치가 있는 내용은 `Potato.CSharp/vibe`에서 빠르게 실험한다.
4. 포팅 결과 중 계속 보관할 만한 것만 `Potato.CSharp/main`에 정리한다.
5. 충분히 검증된 내용만 `Potato/main`으로 반영한다.

이 흐름의 핵심은 모든 결과를 한 곳으로 바로 밀어 넣지 않는 것이다.

학습, 실사용, 포팅 실험은 각각 판단 기준이 다르기 때문에, 한 저장소 안에서 한 번에 관리하려고 하면 오히려 기준이 흐려지기 쉽다.

## 운영하면서 지키려는 원칙

지금 구조에서 계속 유지하려는 기준은 아래와 같다.

- 학습 코드와 실사용 코드를 섞지 않는다.
- `Potato/dev`에는 실사용 후보만 올린다.
- `Potato.CSharp/vibe`는 정제보다 속도와 탐색을 우선한다.
- 레포 간 이동은 통째로 옮기기보다 필요한 내용만 선별해서 반영한다.
- 문서와 디렉터리 구조는 주기적으로 다시 정리한다.

아직 고정된 완성형 운영 방식이라기보다는, 저장소 성격이 뒤섞이지 않도록 기준을 세워두는 쪽에 가깝다.

## 정리

짧게 말하면 지금 구조는 이렇게 이해하면 된다.

- `Potato.Study`는 배우고 실험하는 곳
- `Potato`는 실제로 유지할 코드를 쌓는 곳
- `Potato.CSharp`는 C# 포팅을 시험하고 정리하는 곳

결국 이 구성의 목적은 저장소를 많이 만드는 데 있는 게 아니라, 성격이 다른 작업들을 억지로 한 군데에 섞지 않기 위한 분리라고 보면 된다.
