이전에 공부하면서 처음 Lock Free 알고리즘을 알게 되었을 때, 그런 생각이 들었다. 

> "둘 다 `Interlocked` (또는 CAS - Compare-And-Swap) 같은 원자적(Atomic) 연산을 사용해서 구현하는데, 왜 하나는 Lock'이라는 이름이 붙어있고, 다른 하나는 'Lock-free(락이 없다)'라고 부르는 걸까?"

그 부분에 대해서 정리해보려 합니다.
***

## 0. Interlocked 함수와 원자적(atomic)이란 무엇인가?
*   **원자적(Atomic) 이라는 말은** '더 이상 쪼갤 수 없다'는 뜻으로, 다른 스레드가 중간 상태를 볼 수 없도록, 하나의 끊기지 않는 동작처럼 보장되는 연산을 의미 합니다. 


예를 들어 이런 코드를 보죠.
```cpp
int counter = 0;
counter++;
// 실제로는:
// 1. counter 값을 레지스터에 읽기  (LOAD)
// 2. 레지스터 값에 +1              (ADD)
// 3. 결과를 counter에 다시 쓰기    (STORE)
```

![레이스 컨디션 예시](/projects/spinlock-vs-lockfree/race-condition.svg)

이 세 단계 사이에 다른 스레드가 끼어들면 결과가 망가지게 됩니다.

**원자적 연산은 이 세 단계를 하드웨어 레벨에서 하나로 묶어버려** 중간에 끼어들 틈 자체를 없앱니다.

**Interlocked 함수는** CPU 레벨에서 원자성을 하드웨어적으로 보장하는 **원자적 연산 API** 입니다. 

**CAS (Compare-And-Swap):** 가장 핵심적인 도구로, `old` 값을 확인하여 예상과 일치하면 `new`로 교체하고, 다르면 실패하여 다시 시도할 기회를 줍니다.

💀 원자적 연산 하나가 안전하다고 해서 전체 알고리즘이 안전하다는 뜻은 아닙니다.

Interlocked는 설계의 재료일 뿐입니다.

---

## 1. 락(Lock)은 무엇인가?
락(lock)은

“공유 자원에 대해 특정 코드 구간 전체를 한 번에 한 스레드만 실행하게 만드는 장치”
입니다.

핵심은 **배타적 소유권(다른 스레드의 관여를 허락치 않음)**입니다.

예를 들면:

```cpp
lock();
공유 데이터 읽기
공유 데이터 수정
연결 관계 갱신
unlock();
```
이 구간 안에서는 다른 스레드가 못 들어옵니다.

즉 락은:

* 누군가가 소유하고 있고

* 다른 스레드는 기다려야 하며

* 락을 잡은 쪽이 끝나야 다음 스레드가 들어갈 수 있는 구조 입니다.

> ### 1-1. 스핀락도 락이다.
>
> 스핀락은 이름 그대로 락입니다. 다만 대기 방식이 다릅니다.
>
> * Mutex: 못 들어가면 잠들거나 스케줄링 양보
>
> * SpinLock: 못 들어가면 계속 반복하면서 확인
>
> 즉 스핀락은 **잠들지 않고 바쁘게 기다리는 락 (busy wait)** 입니다.  
>
> 본질은 여전히 락이죠.

---

## 2. Blocking vs Non-blocking

**Blocking:**은 쉽게 말하면,

**내 진행이 다른 스레드 때문에 멈출 수 있는 구조** 입니다.

락을 생각하면 쉽습니다.

```text
A가 락을 잡음
B가 들어가려 함
B는 A가 끝날 때까지 못 감
```
이건 blocking입니다. 즉 내 진행이 락 소유자에게 의존합니다.

**Non-blocking:** 자원을 즉시 쓸 수 없다면 기다리지 않고 바로 제어권을 반환받는 방식입니다. 즉 **특정 스레드가 락을 소유해서 나머지를 세워두는 구조가 아닙니다.**

예를 들어 CAS 기반 알고리즘에서는:

```text
1. 내가 상태를 바꾸려 시도함
2. 다른 스레드가 먼저 바꿨으면 실패
3. 실패하면 최신 상태를 다시 읽고 재시도
```
여기에는 **“누가 락을 풀어주길 기다리는 구조”** 가 없습니다. 

> * 실패는 할 수 있음
> * 재시도는 할 수 있음
> * 하지만 락 소유자 때문에 멈춘 건 아님

이게 non-blocking의 핵심입니다.

---

## 3. 보장되는 안전한 지점 (원자적으로 보호되는 구역)
이 질문을 바꿔 말하면 **"어디까지를 안전하다고 보장하느냐?"** 입니다.

*   **락 기반:** `Lock`과 `Unlock` 사이의 모든 코드 구간이 통째로 원자적으로 보호받습니다.
```cpp
lock();
// 이 구간 전체는 안전
// 다른 스레드가 절대 끼어들 수 없음
unlock();
```
*   **락프리 기반:** 데이터의 상태가 유효한지 Interlocked 연산과 루프를 통해 확인하는 시점마다 안전이 보장됩니다. 

```cpp
old = head;
node->next = old;
CAS(&head, old, node);
```

여기서 진짜 원자적인 건 마지막 CAS입니다.

앞의 읽기와 계산은 보호 구간이 아닙니다.

다른 스레드가 그 사이에 상태를 바꿀 수 있습니다.

그래서 lock-free는:

**“구간 보호”** 가 아니라 **“상태 전이 한 점”** 을 원자적으로 만든다고 보는 게 맞습니다.


## 4. 그래서 무슨 차이임?


| 구분 | 락 (Lock / Mutual Exclusion) | 락 프리 (Lock-free) |
| :--- | :--- | :--- |
| **핵심 메커니즘** | **소유권 기반** (락을 획득해야 진입) | **경쟁 기반** (누구나 시도, 실패 시 재시도) |
| **보호 범위** | **구간(임계구역) 보호** (여러 줄의 코드) | **지점(CAS) 보호** (단일 메모리 갱신 시점) |
| **실패 시 행동** | **대기 (Wait)** (Spin 또는 Sleep) | **재시도 (Retry)** (상태 확인 후 다시 시도) |
| **성격** | **비관적(Pessimistic)** (충돌이 날 거라 가정) | **낙관적(Optimistic)** (충돌이 안 날 거라 가정) |
| **장점** | 설계가 직관적이고 구현이 비교적 쉬움 | 데드락(Deadlock)이 없고 컨텍스트 스위칭 비용 적음 |
| **단점** | 데드락 위험, 락 홀더가 멈추면 전체 정체 | 과도한 경쟁 시 CPU 점유율 상승 (Livelock[^1] 가능성) |

[^1]: Livelock: 프로세스들이 계속해서 CPU를 소모하며 상태를 바꾸지만, 실질적인 결과물은 내지 못하는 상태 (동적 상태)

---

### 💡 최종 정리

결국 "락이 있냐 없냐"의 이름 차이는, **"어떤 스레드의 행동(지연, 중단)이 다른 스레드의 진행을 가로막을 수 있는 시스템인가(Blocking) 아닌가(Non-blocking)?"** 라는 차이에서 비롯된 것입니다.

---

> ### 가장 좋은 것은 공유 자체를 줄이는 것이다. (이게 제일 중요)
> 예를 들면:
> ```text
> 룸 단위 단일 스레드 처리
> actor / event loop 구조
> owner thread 명확화
> 메시지 패싱
> ```
>
> 이렇게 하면 애초에 락이 거의 필요 없어집니다. 
>
> 하지만 위에 언급된 예시를 구현하려면 자료 구조에 락과 락프리 중 하나를 사용해야겠죠?
