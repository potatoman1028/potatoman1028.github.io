이전에 공부하면서 처음 Lock Free 알고리즘을 알게 되었을 때, 이런 생각이 들었습니다. 

> "둘 다 `Interlocked` (또는 CAS - Compare-And-Swap) 같은 원자적(Atomic) 연산을 사용해서 구현하는데, 왜 하나는 Lock'이라는 이름이 붙어있고, 다른 하나는 'Lock-free(락이 없다)'라고 부르는 걸까?"

이 글에서는 **스핀락은 락의 한 종류**라는 점을 먼저 분명히 해두고, 그 위에서 **lock 계열과 lock-free가 무엇이 다른지**를 정리해보려 합니다.
***

## 0. Interlocked 함수와 원자적(atomic)이란 무엇인가?
*   **원자적(Atomic)** 이라는 말은 연산이 더 이상 쪼개지지 않는다는 뜻입니다. 즉, 다른 스레드가 중간 상태를 볼 수 없고, 하나의 끊기지 않는 동작처럼 보이도록 보장되는 연산을 말합니다.


예를 들어 이런 코드를 보죠.
```cpp
int counter = 0;
counter++;
// 실제로는:
// 1. counter 값을 레지스터에 읽기  (LOAD)
// 2. 레지스터 값에 +1              (ADD)
// 3. 결과를 counter에 다시 쓰기    (STORE)
```

![레이스 컨디션 예시](/projects/spinlock-vs-lockfree/race-condition.svg)

이 세 단계 사이에 다른 스레드가 끼어들면 결과가 망가질 수 있습니다.

**원자적 연산은 이런 읽기-수정-쓰기 과정을 하드웨어 수준에서 하나의 끊기지 않는 동작처럼 보이게 만들어** 중간에 끼어들 틈을 없앱니다.

**Interlocked 함수는** 이런 원자적 연산을 제공하는 API입니다.

**CAS (Compare-And-Swap):** 가장 핵심적인 도구로, 현재 값이 내가 예상한 `old`와 같으면 `new`로 교체하고, 다르면 실패하여 다시 시도할 기회를 줍니다.

원자적 연산 하나가 안전하다고 해서 전체 알고리즘이 자동으로 안전해지는 것은 아닙니다.

Interlocked는 완성된 답이라기보다, 동시성 알고리즘[^2]을 구성하는 **재료**에 가깝습니다.

---

## 1. 락(Lock)은 무엇인가?
락(lock)은 **공유 자원에 접근하는 코드 구간 전체를 한 번에 한 스레드만 실행하도록 만드는 장치**입니다.

핵심은 **배타적 소유권(다른 스레드의 관여를 허용하지 않음)**입니다.

예를 들면:

```cpp
lock();
공유 데이터 읽기
공유 데이터 수정
연결 관계 갱신
unlock();
```
이 구간 안에서는 다른 스레드가 들어올 수 없습니다.

즉 락은:

* 누군가가 소유하고 있고

* 다른 스레드는 기다려야 하며

* 락을 잡은 쪽이 끝나야 다음 스레드가 들어갈 수 있는 구조입니다.

> ### 1-1. 스핀락도 락이다.
>
> 스핀락은 이름 그대로 락입니다. 다만 대기 방식이 다릅니다.
>
> * Mutex: 못 들어가면 잠들거나 스케줄링 양보
>
> * SpinLock: 못 들어가면 계속 반복하면서 확인
>
> 즉 스핀락은 **잠들지 않고 바쁘게 기다리는 락 (busy wait)** 입니다.  
>
> 본질은 여전히 락입니다.
>
> 따라서 스핀락에서도 **락 경합(lock contention)** 은 그대로 발생합니다.
>
> 차이는 경합이 생겼을 때 스레드가 **잠들어서 기다리느냐**, 아니면 **CPU를 쓰면서 반복 확인하느냐**에 있습니다.
>
> 즉, 스핀락과 lock-free의 차이는 단순히 "기다리느냐 마느냐"가 아니라, **락 소유권 자체로 임계구역[^3]을 제어하느냐**에서 갈립니다.

---

## 2. Blocking vs Non-blocking

**Blocking**은 쉽게 말하면,

**내 진행이 다른 스레드의 상태에 직접 의존할 수 있는 구조**입니다.

락을 생각하면 쉽습니다.

```text
A가 락을 잡음
B가 들어가려 함
B는 A가 끝날 때까지 못 감
```
이건 blocking입니다. 즉, 내 진행이 락 소유자에게 의존합니다.

**Non-blocking**은 특정 스레드가 멈추거나 지연되더라도, 시스템 전체의 진행이 그 스레드 하나에 묶이지 않도록 설계된 방식입니다.

예를 들어 CAS 기반 알고리즘에서는:

```text
1. 내가 상태를 바꾸려 시도함
2. 다른 스레드가 먼저 바꿨으면 실패
3. 실패하면 최신 상태를 다시 읽고 재시도
```
여기에는 **“누가 락을 풀어주길 기다리는 구조”** 가 없습니다. 

> * 실패는 할 수 있음
> * 재시도는 할 수 있음
> * 하지만 락 소유자 때문에 멈춘 건 아님

정확히 말하면, lock-free는 **모든 스레드가 즉시 성공한다**는 뜻은 아닙니다.

어떤 스레드는 여러 번 실패할 수 있지만, **시스템 전체로 보면 누군가는 계속 전진할 수 있다**는 점이 핵심입니다.

## 3. 보장되는 안전한 지점 (원자적으로 보호되는 구역)
이 질문을 바꿔 말하면 **"어디까지를 안전하다고 보장하느냐?"** 입니다.

*   **락 기반:** `Lock`과 `Unlock` 사이의 임계구역[^3] 전체가 보호됩니다.
```cpp
lock();
// 이 구간 전체는 안전
// 다른 스레드가 절대 끼어들 수 없음
unlock();
```
*   **락프리 기반:** 여러 스레드가 동시에 같은 상태를 읽고 계산할 수 있고, 마지막에 상태를 바꾸는 **원자적 전이 지점**[^4]에서만 충돌 여부가 판정됩니다. 

```cpp
old = head;
node->next = old;
CAS(&head, old, node);
```

여기서 원자적인 것은 마지막 CAS입니다.

앞의 읽기와 계산은 락처럼 통째로 보호된 구간이 아닙니다.

다른 스레드가 그 사이에 상태를 바꿀 수 있습니다.

그래서 lock-free는:

**“구간 전체를 잠그는 방식”** 이 아니라, **“상태가 여전히 내가 예상한 그대로인지 확인하고, 맞으면 갱신하고, 아니면 다시 시도하는 방식”** 에 가깝습니다.


## 4. 그래서 무슨 차이임?

여기서는 혼동을 줄이기 위해 **블로킹 락(예: mutex)**[^5], **스핀락**, **lock-free**를 나눠서 보면 더 명확합니다.

| 구분 | 블로킹 락 (예: Mutex) | 스핀락 (Spinlock) | 락 프리 (Lock-free) |
| :--- | :--- | :--- | :--- |
| **핵심 메커니즘** | **소유권 기반** (락을 획득해야 진입) | **소유권 기반** (락을 획득해야 진입) | **경쟁 기반** (누구나 시도, 실패 시 재시도) |
| **경합이 발생하면** | 락이 풀릴 때까지 잠들거나 스케줄링 양보 | 락이 풀릴 때까지 계속 반복 확인 | 누가 먼저 성공하면 나머지는 최신 상태를 읽고 재시도 |
| **보호 방식** | **임계구역 보호** (구간 전체를 배타적으로 실행) | **임계구역 보호** (구간 전체를 배타적으로 실행) | **상태 전이 보호** (검사 후 원자적으로 갱신, 실패 시 재시도) |
| **경합의 종류** | **락 경합(lock contention)**[^6] | **락 경합(lock contention)**[^6] | **락 경합은 없지만 CAS 충돌과 재시도 경쟁은 있음** |
| **CPU 사용 특성** | 대기 중 CPU를 덜 씀 | 대기 중 CPU를 계속 사용함 | 경합이 심하면 재시도로 CPU 사용량이 커질 수 있음 |
| **성격** | **비관적(Pessimistic)** | **비관적(Pessimistic)** | **낙관적(Optimistic)** |
| **장점** | 구현이 직관적이고 긴 임계구역에도 대응하기 쉬움 | 매우 짧은 임계구역에서 문맥 전환 비용을 아낄 수 있음 | 데드락(Deadlock)이 없고 락 소유자에 의해 직접 멈추지 않음 |
| **단점** | 락 홀더가 지연되면 전체가 같이 기다림 | 경합이 길어지면 CPU를 낭비하고, 락 홀더가 안 풀면 계속 돎 | 구현 난도가 높고, 과도한 경쟁 시 CPU 점유율 상승 (Livelock[^1] 가능성) |

[^1]: Livelock: 프로세스들이 계속해서 CPU를 소모하며 상태를 바꾸지만, 실질적인 결과물은 내지 못하는 상태 (동적 상태)
[^2]: 동시성 알고리즘: 여러 스레드가 동시에 같은 데이터나 자원에 접근해도 올바르게 동작하도록 설계된 알고리즘.
[^3]: 임계구역(Critical Section): 여러 스레드가 동시에 실행하면 데이터가 깨질 수 있어서, 한 번에 하나의 스레드만 들어가야 하는 코드 구간.
[^4]: 원자적 전이 지점: 공유 상태를 실제로 바꾸는 순간으로, 그 변경이 성공할지 실패할지가 한 번에 결정되는 지점.
[^5]: 블로킹 락(Blocking Lock): 락을 바로 얻지 못하면 스레드가 잠들거나 스케줄러에 의해 대기 상태로 넘어가는 락. 대표적으로 mutex가 여기에 속합니다.
[^6]: 락 경합(Lock Contention): 여러 스레드가 같은 락을 동시에 얻으려 해서, 일부 스레드가 대기하게 되는 상황.

---

### 💡 최종 정리

결국 이름의 차이는 단순히 **원자적 연산을 쓰느냐**가 아니라,

**공유 자원의 접근을 락 소유권으로 제어하느냐**, 아니면 **원자적 상태 전이와 재시도로 진행을 보장하느냐**의 차이에서 나옵니다.

스핀락은 여전히 **락의 한 종류**이고, lock-free는 락 소유권 없이도 동시성을 다루도록 설계된 **non-blocking 계열**입니다.

---

> ### 가장 좋은 것은 공유 자체를 줄이는 것이다. (이게 제일 중요)
> 예를 들면:
> ```text
> 룸 단위 단일 스레드 처리
> actor / event loop 구조
> ```
>
> 이렇게 하면 애초에 락이 거의 필요 없어집니다.
>
> 하지만 공유를 완전히 피할 수 없는 자료 구조 수준으로 내려오면,
> 그때는 lock 기반으로 단순하게 갈지, lock-free로 복잡도를 감수하고 다른 특성을 가져갈지를 선택해야 합니다.
